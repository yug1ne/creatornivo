README GOAL

Create a README that helps a new visitor quickly understand:

- what the project is;
- what problem it solves;
- who it is for;
- its current development status;
- what is already implemented;
- how to install it;
- how to configure it;
- how to run it;
- how to use it;
- how to test it;
- how to deploy it when relevant;
- where to find deeper documentation;
- how to report problems;
- how to contribute;
- what license applies.

The README should make the shortest successful path easy to find.

A developer should not need to read the entire file before locating the installation and quick-start instructions.

ACCURACY RULES

1. Use only information supplied in the prompt.

2. Do not invent:

- features;
- commands;
- package names;
- environment variables;
- ports;
- URLs;
- API endpoints;
- CLI arguments;
- configuration options;
- dependencies;
- supported operating systems;
- supported browsers;
- compatibility;
- benchmarks;
- security properties;
- licenses;
- contributors;
- maintainers;
- releases;
- version numbers;
- deployment options;
- test coverage;
- badges;
- customer logos;
- usage examples;
- expected output.

3. Do not assume standard commands merely from the project language.

For example, do not automatically write:

npm install
npm run dev
pip install
cargo build
docker compose up

unless the command or tooling was supplied.

4. Do not infer the package manager solely from a lock file unless the repository information explicitly confirms its intended use.

5. Do not invent environment variables to make an example look complete.

6. Do not invent placeholder API keys that resemble real credentials.

7. Do not state that the project is:

- production-ready;
- secure;
- scalable;
- fast;
- lightweight;
- enterprise-ready;
- privacy-friendly;
- accessible;
- battle-tested;
- actively maintained;
- cross-platform;
- zero-configuration;

unless supported by supplied information.

8. Do not describe planned functionality as currently available.

9. Clearly label:

- experimental features;
- beta functionality;
- deprecated options;
- incomplete documentation;
- known limitations;
- planned work.

10. When required information is missing, insert a visible marker:

[INFORMATION REQUIRED: exact installation command]

Do not replace missing information with assumptions.

INSTALLATION

Adapt installation instructions to the project type.

For a library or package:

- show the confirmed package-manager command;
- provide import or initialization code;
- mention peer dependencies;
- explain version compatibility.

For an application:

- show clone, dependency, configuration, and startup steps;
- explain local services;
- identify the working directory.

For a CLI:

- show supported installation methods;
- show how to verify the installed version;
- explain PATH requirements only when relevant.

For a binary release:

- explain how to choose the correct operating-system and architecture build;
- include checksum or signature verification only when supplied.

For Docker:

- distinguish image pull, image build, and Docker Compose workflows;
- explain required volumes, ports, and environment variables;
- do not expose production secrets.

For a GitHub Action:

- show a minimal workflow example;
- pin to the confirmed version;
- list permissions, inputs, outputs, and required secrets.

Do not include an installation method that is not officially supported by the project.

Do not put real secrets in examples.

USAGE

- expected result;

- use confirmed public APIs;
- include necessary initialization;
- avoid deprecated methods;
- include error handling when useful;
- keep examples executable or clearly label them as pseudocode.

Do not invent output.

When output varies, explain that it is illustrative.
